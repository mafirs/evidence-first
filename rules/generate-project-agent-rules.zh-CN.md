# 为当前仓库生成项目级 Agent 规则

这是一份通用 Prompt，适用于 Codex、Claude Code 或其他能读取仓库文件的 AI coding agent。目标是让 agent 实读当前仓库，并生成项目级 agent 规则文件，例如 `AGENTS.md`、`CLAUDE.md`、`.cursorrules` 或团队指定的等价文件。

你的任务：实读当前仓库，结合下方【默认工作习惯不变量】，生成一份项目级 agent 规则文件。
默认目标文件为仓库根目录 `AGENTS.md`；如果用户指定 `CLAUDE.md`、`.cursorrules` 或其他等价文件，以用户指定为准。
本轮只允许创建 / 覆盖用户指定的项目级 agent 规则文件；未指定时只允许创建 / 覆盖仓库根目录 `AGENTS.md`。不许改动任何其他文件，不许执行会改变项目状态的命令（只读命令如 `git log`、`ls`、`cat` 允许）。

【第一步：调研仓库 — 必须实读】
━━━━━━━━━━━━━━━━━━━━

依次读取并提取事实（存在才写，不存在跳过，禁止编造）：

1. 包管理与命令：`package.json` / `pnpm-workspace.yaml` / `Makefile` / `turbo.json` 等
   → 实际使用的包管理器、dev / build / test / lint / typecheck 的真实命令。
2. 工程结构：顶层目录布局、monorepo 的 packages / apps 划分、各模块职责一句话。
3. 代码约定：lint / format 配置（eslint / biome / prettier）、tsconfig 严格度、实际代码中的主导模式（API 层、错误处理、状态管理等）——从 2-3 个代表性源文件中归纳，标注你读了哪几个文件。
4. 测试：测试框架、测试文件命名与位置、如何跑单个测试。
5. Git 与 CI：分支模型线索（`git log --oneline -20`）、commit message 实际风格、`.github/workflows` 里 CI 会跑什么检查。
6. 已有 agent 文件：若已存在 `AGENTS.md` / `CLAUDE.md` / `.cursorrules` / Cursor rules，读取并在新文件中合并仍然成立的内容，过时内容丢弃并在最后汇报中说明。

调研结束后，先列出“我实际打开过的文件清单”，再开始写规则文件。
清单之外的文件不允许作为规则文件中任何断言的依据。
无法通过实读确认的内容，要么不写，要么标注 `[假设-需验证]`。

【第二步：项目级规则文件结构 — 按此组织】
━━━━━━━━━━━━━━━━━━━━

## Scope
一句话：本文件适用于本仓库；与工具全局规则冲突时，以本项目级规则为准。如果当前工具支持更近层级的规则文件，则以更近者为准。

## Commands
从第一步第 1、4 条提取真实命令。只写会被实际用到的。

## Architecture
第一步第 2 条的结果。每个模块一行，超过 10 行就砍。

## Code Conventions
第一步第 3 条的结果。只写“本项目实际在用”的约定，禁止写“保持代码整洁”这类空话。

## Working Principles
将下方【默认工作习惯不变量】中的 1-6 改写为简洁规则放入此节。

## Safety
将不变量中的 7-11 改写放入此节。

## Verification
将不变量中的 12-13 改写，并结合本仓库真实的 test / lint / typecheck 命令具体化。例如不要写“跑相关测试”，要写“运行 `pnpm test -- <path>` 跑最窄相关测试”。

## Communication
将不变量中的 14-16 改写放入此节。

━━━━━━━━━━━━━━━━━━━━
【默认工作习惯不变量】
━━━━━━━━━━━━━━━━━━━━

工作原则：
1. 先读后写：修改任何文件前必须先实读；方案 / 分析中不允许出现未读过的文件。
2. 现状断言分级：对代码现状的描述区分 `[已读确认]`（基于实读）与 `[假设-需验证]`（基于推断），后者必须附“若假设不成立怎么调整”。
3. 最小修改：只改实现需求所必需的代码 + 与本次改动直接相关的安全 / 边界处理（空值、异常、并发、权限）；其他一律不动。
4. 改动分类：每处改动标注 `(a)` 需求必需 / `(b)` 安全健壮性必需 / `(c)` 其他；`(c)` 类不执行，移入“额外发现”。
5. 额外发现不夹带：读代码时发现的需求外问题（bug、隐患、可优化点）单独列出报告，由用户决定是否处理，禁止夹带进本次改动。
6. 不确定先问：关键不确定点（实质影响方案的）先提问，最多 3 个；低风险点按合理假设走并标注。

安全红线：
7. 禁止顺手行为：重命名、调整格式 / 缩进 / 引号、改 import 顺序、删“看起来没用”的代码 / 注释 / 日志、重构、抽函数、改与需求无关的类型 / 接口签名。方案中出现“顺便 / 同时 / 另外建议 / 更规范 / 优化一下”等措辞即视为越界信号。
8. 命令连续失败 2 次即停下报告，禁止绕路（删测试、改断言、skip、放宽类型、`--force`）。
9. 测试失败说明代码错而不是测试错；禁止为变绿而改测试。
10. destructive 操作（删文件、`git reset` / force push、migration、依赖升级）先确认；不回滚用户未要求回滚的改动；不手改生成文件；不提交 secrets。
11. 编译 / 类型检查失败：属于机械失误（import 遗漏、括号不配对、明显拼写）可自行修复，但必须在汇报中标注“自行修复”；属于逻辑问题则如实贴出等用户决定，不要修。

验证与完成标准：
12. 代码改动后必须跑最窄的相关测试；共享代码加跑 lint / typecheck；报告跑了什么命令、是否通过。
13. 无法验证时明说原因 + 残余风险；禁止用“应该没问题”代替验证。

沟通：
14. 默认跟随用户语言；代码、注释、commit message 用英文，除非项目已有其他约定。
15. 汇报结构：改了哪些文件 → 未执行项及原因 → 验证结果 → 自审疑点 → 需用户手动确认的回归点 → 发现但未处理的问题。
16. 提出问题 / 风险时标注置信度：`[核心]`（不解决会真出事）/ `[边缘]`（理论成立但不确定会触发）/ `[凑数]`（不是为了列问题不会主动提）。`[凑数]` 无惩罚；把 `[凑数]` 伪装成 `[核心]` 才扣分。业务影响用人话描述用户会看到什么变化，不要堆代码术语。

━━━━━━━━━━━━━━━━━━━━
【硬性约束】
━━━━━━━━━━━━━━━━━━━━

- 总长度 ≤ 120 行。超了就砍：优先砍 Architecture 和 Code Conventions 的细节，Working Principles / Safety / Verification 不许砍。
- 只写纠错性内容：模型默认就会做对的事不写；写进去的每一行都应该在阻止某个具体错误行为。
- 区分两类内容，不许混淆：仓库事实（命令、结构、约定）必须来自实读；工作习惯来自上方不变量。
- 以下内容禁止写入项目级规则文件：
  - 多阶段工作流（定位 → 路线 → 评审复核循环 → 方案 → 文档审查 → 执行的完整流程编排）——那是按需调用的 prompt / skill，不是常驻指令。
  - 详细输出模板（六区块方案格式、逐条回应格式）——同上。
  - “写高质量代码”“注意边界条件”等无具体指向的空话。
  - 任何你没有实读验证过的仓库事实。

━━━━━━━━━━━━━━━━━━━━
【输出前自检】
━━━━━━━━━━━━━━━━━━━━

1. Commands 里每条命令是否都来自实读的配置文件？
2. 是否有任何断言基于未打开过的文件？有则删除或标 `[假设-需验证]`。
3. 总行数是否 ≤ 120？
4. 是否混入了流程编排或输出模板？有则删除。
5. 逐行问：删掉这一行，agent 会犯什么具体错误？答不上来就删。

完成后汇报：实读文件清单、生成的规则文件路径和行数、合并 / 丢弃旧 agent 文件的情况（如有）、标注了 `[假设-需验证]` 的条目清单。
