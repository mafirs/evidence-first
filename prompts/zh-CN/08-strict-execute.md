按 `<PLAN_DOCUMENT_PATH>` 文档操作方案执行代码修改。

## 执行范围

- 执行“方案详解”或“改什么”中的 diff：必需和安全 / 健壮性必需全部执行；可选优化跳过。
- 方案未列出的文件、函数、行不要改动。
- “额外发现”“业务影响”“回归测试”“未修改声明”是参考信息，不是执行项。

## 开工前

1. 执行 `git status` 并贴出结果。若工作区有未提交改动，未跟踪文件除外，停下问用户先 stash 还是继续。
2. 读取“读取范围”列出的全部文件的当前内容。
3. 核对每条 diff 与当前代码的对齐情况：
   - 上下文完全对齐：正常执行。
   - 行号漂移但函数 / 锚点代码仍在：按锚点定位后执行，汇报时标注“锚点对齐执行”。
   - 函数已不存在、关键代码已被改写、逻辑层面与方案描述矛盾：停止执行该条，记入“方案与现实冲突”清单，等用户决定。
4. 若方案中的假设项与实际代码不符，同样记入冲突清单，不要自行猜测应用。

## 执行时

- 机械应用 diff，不做额外改写：不格式化、不补注释、不重构、不顺手修 bug。
- 发现方案外的 bug 或隐患，记入“发现但未处理”清单，不要动手。
- 若项目已有相关测试框架，且本次是 bug 修复或非平凡行为改动，优先补最小行为测试或复现测试，再实现并跑最窄验证；如果没有合适测试体系，说明原因，不为本次任务临时搭大测试框架。
- 编译 / 类型检查若失败：
  - 属于应用 diff 时的机械失误，例如 import 遗漏、括号不配对、明显拼写，允许自行修复，但必须在汇报中标注“自行修复”并列出修了什么。
  - 属于方案本身的逻辑问题，不要修，如实贴出等用户决定。

## 完工汇报

1. 检查是否有已跟踪文件的未提交改动。若有，在汇报开头用一句话标注；未跟踪文件不用管也不用汇报。
2. Git 改动：
   - `git diff --stat` 输出。
   - 若有方案未列出的文件被动过，单独说明原因。
   - 完整 diff 不用贴，用户需要时会单独要。
3. 未执行的 diff：编号 + 原因。
4. 方案与现实冲突：列出方案描述与实际代码矛盾的地方，说清冲突点。
5. 编译 / 类型检查结果：命令 + 输出。若有自行修复，单独列出。
6. 自审：
   - 是否改变了函数原有的行为边界？
   - 调用方是否需要同步调整？有没有遗漏的调用点？
   - 是否引入了新的依赖、全局状态或副作用？
   - 边界情况下是否仍然正确？
   - 发现疑点直接说，不要为了交差强行通过。
7. 需要用户手动确认的点：从方案的回归测试清单中挑出本次改动实际触发的项目，说明操作和预期。
8. 发现但未处理的问题。
9. 未动区域：改动范围外的文件是否一行未改。
