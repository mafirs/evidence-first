# 04 — 只有现象、没有预设方案时的路线比较

上文是我和你对当前问题现象的讨论。我没有预设解决方向，方案由你来铺。本轮只读不改。

## 先把现象落到根因或杠杆

发散方案前必须做这一步，讨论不能替代读码。

- 上文讨论的是现象，不是结论。先读相关代码，把现象坐实到它实际起源或真正瓶颈，使用 `文件:符号名` 标注，并标 `[已读确认]` 或 `[假设-需验证]`。
- 现象不等于根因。根因若已在讨论中坐实，一句话引用确认点；若只到现象、根因未定或分叉成多个候选，如实列出分支及各自触发条件。此时方案按根因分支组织，禁止默认挑一个根因就开始发散。
- 反锚定：讨论里若已飘出某个倾向方向，那只是症状收集的副产品，不作为起点；解空间仍然全开。

## 发散

要广不要精，先别评优劣。

- 沿光谱尽量铺开：贴现有架构的增量 / 结构性改动 / 借鉴外部实现 / 重新框定问题。
- 每条一行：机制 + 落点。落点使用 `文件:符号名`；需新增则写“新增 + 一句理由”。
- 这一步不套“必须可证最优”“必须复用现有机制”的闸，故意放宽，把非显而易见的也列出来。
- 外部参考给 repo + 文件定位，并点出能否映射回本库；映射不了的标“仅供参考、本库落不了地”。

## 收敛

- 聚类、合并变体，提炼成 2-4 条真正不同的路线。架构落点或机制必须不同，写法/参数差异不算。
- 落不了地、纯噪音的淘汰并一句话说明为什么。
- 每条幸存路线补：它赌的是什么 / 在什么前提下它最优。若挂在不同根因分支下，标清对应哪个分支。

## 对比表

逐路线过完每一维，每格一句证据定位或推理依据。实读标来源，推断写明是推断。

维度：与现有架构契合度 / 侵入面与波及面 / 兼容性破坏 / 后续扩展性 / 实现工作量

## 推荐

不用单一优先级序硬收敛到一个。沿“保守增量 ↔ 结构性重构”排开，给默认倾向和理由，并写清：你若更看重长期扩展性 / 最小波及 / 快速见效，则换哪条。

若根因仍分叉，推荐按分支给：根因是 A 推哪条，根因是 B 推哪条。附反转条件。

## 任务边界与交接

本轮只输出路线对比与推荐；文件级改动清单、diff、实现细节归下一阶段设计 Prompt，不要输出。

末尾汇总一段供我带入设计阶段：选定路线的架构落点 + 关键约束 + 已声明假设，包含选定的根因分支。

## 不确定点

只问会改变根因判断、路线集合或推荐倾向的问题，最多 3 个；其余按最合理假设走并标在对应路线上。
