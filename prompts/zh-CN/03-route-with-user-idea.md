# 03 — 用户已有想法时的路线比较

我已在上文给出：聚焦点、当前不足、我自己的优化想法。本轮只读不改，产出路线对比与推荐。

## 出方案前先验证输入

- 验证“当前不足”是否属实：读相关代码确认，标 `[已读确认]` 或 `[假设-需验证]`。若复现不出，或实际成因与我说的不符，先说清楚再往下，别基于错误前提出方案。
- 评估我的优化想法：技术上成立吗、落点在哪，用 `文件:符号名` 标注，有没有阻断性问题。这一步只评估不站队，不许因为是我提的就默认它对。

## 方案生成纪律

- 解决范围锁死在“消除上述不足”，不锁死在“实现我的想法”。我的想法是候选之一，不是约束。
- 我的想法默认作为一个候选方案，和其它方案同表对比、平等排序。它好就推它；它被别的方案打败就如实说它败在哪一维、差多少。禁止礼貌性采纳，也禁止无视。
- 若上一步发现它有阻断性问题，改为单列说明它为什么出局，不必硬塞进对比表。
- 真多样性：方案之间在架构落点或实现机制上不同；数量 1-3。只有一条合理路线就说明为什么，禁止凑数。
- 每个方案锚定现有机制或扩展点，使用 `文件:符号名` 标注；凭空新建一套的，须说明现有机制为什么不可复用。
- 允许发散检索外部参考，引用 repo + 文件定位；外部方案必须映射回本库具体落点，落不了地的不列。
- 每个方案声明“在什么前提下它是最优解”，答不出的不出现。

## 对比表

逐方案过完每一维，每格一句证据定位或推理依据。实读标来源，推断写明是推断。

维度：与现有架构契合度 / 侵入面与波及面 / 兼容性破坏 / 后续扩展性 / 实现工作量

## 推荐裁决

固定优先级序推导：架构契合 > 波及面小 > 兼容性 > 扩展性 > 工作量。推荐只能由对比表和该优先级推出，禁止表外理由。

如果我的想法不是首选，明说它输在哪、差多少，别绕。附反转条件：什么前提下推荐换成另一个。

## 任务边界与交接

本轮只输出路线对比与推荐；文件级改动清单、diff、实现细节归下一阶段设计 Prompt，不要输出。

裁决末尾汇总一段供我带入设计阶段：选定路线的架构落点 + 关键约束 + 已声明假设。

## 不确定点

只问会改变排序或推荐的问题，最多 3 个；其余按最合理假设走并标在对应方案上。
