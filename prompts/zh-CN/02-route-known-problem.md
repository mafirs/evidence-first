# 02 — 已知问题后的路线比较

基于本会话已完成的定位结论进入方案阶段。本轮只读不改：只产出路线对比与推荐，不输出 diff、文件改动清单或实现细节。

## 方案生成

- 数量 1-3，由问题本身决定。只有一条合理路线就只给一条，并说明为什么没有第二条；禁止凑数。写法或参数差异算同一方案的变体，独立方案必须在架构落点或修复机制上真不同。
- 每个方案必须讲清三件事：
  1. 消除的是根因还是触发条件，标 `[已读确认]` 或 `[假设-需验证]`。
  2. 为什么提它：核心取舍是什么，用一句话点出它拿什么换什么。
  3. 是否契合本项目：依托哪个现有机制落地，使用 `文件:符号名` 标注；契合点和别扭点都说，别只挑好的讲。
- 怀疑是依赖或上游已知问题时，检索其 issue 与官方修复并附链接；查不到写“未查到”，不许编。
- 止血方案若存在，单独列出适用条件；止血方案不参与排序与打档。

## 对比表

逐方案过完每一维，每格一句证据定位或推理依据。实读的标来源，推断的写明是推断。

维度：根治性 / 回归风险 / 侵入面 / 兼容性影响 / 实现工作量

## 推荐程度

给每个方案打一档并说明理由：`[首选]` / `[可选]` / `[不推荐]`。

评级只能由对比表推导，优先级序固定：根治性 > 回归风险小 > 侵入面小 > 工作量小。禁止引入表外理由。

标 `[首选]` 的方案必须附反转条件：在什么前提下它会让位给另一个方案。

## 交接

裁决末尾汇总一段，供我选定路线后直接带入设计阶段：该路线的架构落点 + 关键约束 + 已声明假设。

## 不确定点

只问会改变排序或推荐档位的问题，最多 3 个；其余按最合理假设走，标在对应方案上。
