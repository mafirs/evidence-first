# 06 — 大修方案

你现在只允许输出修改方案，严禁直接修改、创建、删除任何项目文件，严禁执行会改变项目状态的命令。方案由用户手动确认后执行，你只负责出施工图。

请基于当前代码给出修改方案。

## 前置要求

### 1. 先读代码，再出方案

出方案前，必须先实际读取相关代码。未读过的文件不允许出现在方案中。

### 2. 不确定先提问

如果需求、代码逻辑、数据结构、接口行为、边界条件存在关键不确定点，先提问，不要自行假设后直接出方案。

最多提 3 个关键问题，只提会实质影响方案的问题。低风险点可直接按假设走，但必须在方案中标注。

### 3. 最小修改原则

只改为实现本次需求所必需的代码，以及与本次改动直接相关的安全性、健壮性、边界处理。

发现需求之外的问题，不要在方案中直接修改，放到“额外发现”区告诉用户，由用户决定是否本次一起处理。

禁止顺手做以下事情：重命名变量 / 函数 / 文件，调整格式缩进引号，改动 import 顺序，删除“看起来没用”的代码 / 注释 / 日志，重构，抽函数，修改与本需求无关的类型 / 接口签名。

## 输出结构

严格按以下顺序和分区输出，不要混合。

### 一、读取范围

列出本次实际读取过的：

- 文件路径
- 关键函数 / 类 / 模块名
- 涉及的行号区间

未在此列表出现的文件，后续不允许出现在方案中。

### 二、方案详解

按每个文件一个小节组织，每个小节包含：

**文件路径**：完整路径

**定位方式**：如何找到修改位置，例如函数名、类名、锚点代码

**Diff**：unified diff 格式，带足够上下文

**改动分类**：对本文件的每一处改动，标注属于：

- (a) 实现需求所必需
- (b) 为保证本次改动的安全性 / 健壮性 / 边界处理所必需
- (c) 其他

标为 (c) 的改动必须移除，或移到“额外发现”区。

**现状断言来源**：本小节中对代码现状的描述，逐条标注：

- `[已读确认]` 基于实际读取的代码
- `[假设-需验证]` 基于推断，并说明若假设不成立，方案需如何调整

**为什么这样改**：说明该方案的必要性。

### 三、业务影响

用人话写，不要代码术语：

- 这次改动在业务上做了什么
- 哪些页面 / 功能 / 用户行为会发生变化
- 哪些原有行为保持不变
- 改完后用户应该看到的实际结果

### 四、回归测试清单

除了新功能本身，列出需要手动确认没坏的原有功能：

- 具体点哪个页面 / 按钮 / 流程
- 输入什么数据
- 预期看到什么结果
- 为什么这个功能可能受本次改动影响

### 五、额外发现

本次读代码过程中发现的、与需求无关的问题：

- 问题位置
- 问题描述
- 潜在风险
- 建议处理方式
- 是否建议本次一起修，及理由

这些不要夹带进“方案详解”的 diff 里，单独列出等用户决定。

### 六、未修改声明

明确列出本次方案未触及、但看起来相关的文件 / 模块 / 功能，让用户知道边界在哪。

## 语言约束

方案中若出现“顺便 / 同时 / 另外建议 / 更好的做法是 / 更规范 / 更清晰 / 优化一下 / 建议重构”这类措辞，视为越界信号，对应改动必须移入“额外发现”区，不得出现在“方案详解”里。

## 最终自检

输出前自问并回答：

1. 是否可以直接给方案？还是需要先提问？
2. “方案详解”里的每一处 diff，分类是 (a)(b) 还是 (c)？(c) 是否已移除或移入“额外发现”？
3. “读取范围”列出的文件，是否覆盖了方案中涉及的所有文件？有没有未读却出现在 diff 里的？
4. “现状断言来源”是否每条都标了 `[已读确认]` 或 `[假设-需验证]`？
5. “回归测试清单”是否包含了所有可能受影响的原有功能？

如需提问，只输出问题，不要输出方案。
