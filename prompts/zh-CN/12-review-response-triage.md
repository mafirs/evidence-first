# 12 — 对另一个 AI 批评的复核与决策

下面是另一个 coding agent 对你方案思路的对抗评审：

```text
<PASTE_REVIEW_HERE>
```

## 你的任务

不要顺着对方的话展开。把这份评审转化成下一版正式方案前的决策清单。

## 输出顺序

1. 思路变化总结。
2. 逐条回应，必须有证据或依据。
3. 归档项，一句话带过低价值条目。

注意：执行顺序是先完成复核，再生成总结；输出顺序仍然必须先展示“思路变化总结”。

## 一、思路变化总结

逐项填写，不允许用“整体符合预期”“基本不变”这类话概括。

1. 最初思路保留了什么？
2. 经过评审后，思路新增了什么？
3. 经过评审后，思路删掉或降级了什么？
4. 对方有没有提出之前没考虑到、且确实成立的问题？有就列出；没有写“无”并说明为什么。
5. 对方有没有钻牛角尖？有就指出是哪几条，为什么不影响当前思路。
6. 有没有需要用户拍板的产品语义或取舍？格式：`[决策点] — 选项 A vs 选项 B — 我的倾向是 X，需要用户确认`。没有写“无”。
7. 有没有如果不改会影响真正上线、或给后续方案挖坑的点？有就列出文件:行号或模块名；没有写“无”并说明。

## 二、逐条回应

必须复核以下条目：

- 对方标为 `[核心]` 的问题。
- 对方标为 `[对话+实读冲突]` 的工程前提。
- 对方标为 `[agent 没说我读后补充]` 且影响当前思路的代码事实。
- 对方提到的“更小的路”。
- 对方路线比较结论为“有更适合路线”时，该路线及其依据。
- 对方提到的产品语义 / 用户拍板项。

`[边缘]` 条目若可能改变正式方案也要复核；否则进归档项。`[凑数]` 默认归档，除非你认为对方低估了它。

对以上每条判断：属实 / 部分属实 / 不属实 / 无法判断。

规则：

- 无论判断为属实、部分属实或不属实，都必须给出依据：代码证据、对话证据或产品语义证据。
- 没有打开对应文件就不能得出“不属实”结论，此时写“无法判断”。
- 选择“无法判断”时，必须说明缺少什么证据；能通过读代码或对话上下文判断的，不允许写“无法判断”。

每条格式：

对应对方第 X 条：[简述]

- 所属层级：代码架构层 / 功能层 / 用户体验层
- 复核结论：属实 / 部分属实 / 不属实 / 无法判断
- 表态：采纳 / 部分采纳 / 不采纳 / 需要用户拍板
- 处理结论：必须改思路 / 方案阶段展开 / 暂不处理 / 对方判断不成立 / 需要用户拍板
- 紧急度：高 / 中 / 低
- 用户感知严重度：高 / 中 / 低
- 理由：限 3 句话

压缩规则：

- 逐条回应最多保留 6 条；每条理由最多 3 句。其余进归档项。
- 表态为“采纳”且理由显而易见时，理由压缩到 1 句。
- 多条重复批评合并，保留原编号。
- `[凑数]` / `[边缘]` 且不影响方案的条目，直接进归档项。

## 三、归档项

每条一句话：

第 X 条 — `[凑数 / 不属实 / 已覆盖]` — [一句话说明为什么不处理]

## 禁止行为

- 禁止写“加测试 / 加日志 / 加注释 / 更严谨 / 注意边界”等无具体指向的话。
- 禁止给具体代码实现。
- 禁止预演尚未确定的实现细节。
- 禁止在未打开文件的情况下对代码现状做确定性表述。
