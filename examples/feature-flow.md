# Feature Flow Walkthrough

这个示例适合你已经有一个实现想法，但不确定它是不是正确路线时阅读。

示例里的产品、接口、文件名都是虚构的，只用于展示工作流。不要把真实业务数据、客户信息、密钥或私人对话提交进仓库。

## 虚构场景

你在一个文档协作应用里想优化搜索体验：

> 当前搜索只匹配标题。你想新增"搜索正文内容"的能力。

你的初始想法是：

> 直接在前端把所有文档内容拉下来，然后在浏览器里做全文搜索。

你担心这个想法可能有性能和权限问题，但还没有看代码，也不知道项目里是否已有搜索 API 或索引机制。

## 第一步：用 03-route-with-user-idea 让 Codex 挑战你的想法

先写清楚当前不足、你的想法和担心，加分隔线，然后复制 `prompts/zh-CN/03-route-with-user-idea.md` 的完整内容。

```text
当前不足：搜索框只能搜文档标题，搜不到正文里的关键词。

我的想法：进入页面时拉取所有文档正文，在前端做全文搜索。

我的担心：可能会导致首屏慢，也可能暴露用户不该看到的文档内容。

本轮只读不改。请先验证当前不足是否属实，再评估我的想法是否成立。我的想法只是候选方案，不是硬约束。

————————
[粘贴 prompts/zh-CN/03-route-with-user-idea.md 的完整内容]
```

Codex 的输出应该包含：

- 当前不足是否属实（读代码确认）。
- 你的想法技术上是否成立，有没有阻断性问题。
- 你的想法与其他路线的对比表。
- 推荐路线和反转条件。

**停下的信号**：Codex 因为是你提的想法就直接采纳，没有横向比较——停下，让它把你的想法作为候选方案之一，与其他路线平等对比。

## 第二步：把 Codex 的讨论上下文发给 Claude 做早期评审

当路线看起来可行，但你想在出正式方案前让 Claude 提前挑刺时，把你和 Codex 的对话上下文复制给 Claude，然后使用 `prompts/zh-CN/09-early-idea-review-with-code.md`。

把模板的 `<PASTE_CONVERSATION_CONTEXT_HERE>` 替换为你和 Codex 的关键对话内容。

Claude 的输出应该包含：

- 原始诉求和当前思路方向的复述。
- Claude 实际打开过的文件列表。
- 对 Codex 描述的代码现状的逐条验证，标明是否一致。
- 是否有更小的路线（如有，给出具体文件:行号）。
- 结论：有阻塞性发现，或可以推进到方案阶段。

**停下的信号**：Claude 的批评没有代码证据，只说"可能会有问题"——要求它给出具体文件和行号，或者承认这是推断而非读代码后的结论。

如果 Claude 没有代码访问能力，改用 `prompts/zh-CN/10-early-idea-review-from-chat.md`，但这条路线通常不如 09 可靠。

## 第三步：让 Codex 出方案

路线确认后，根据改动范围在同一个 Codex 对话里选择 05 或 06。

```text
我选择路线：复用后端搜索 API，在服务端扩展正文匹配，并保持现有权限过滤。

请基于上面的讨论给出方案，不要直接改代码。

————————
[粘贴 prompts/zh-CN/06-large-plan.md 的完整内容]
```

Codex 的输出应该包含：

- 读了哪些文件和关键位置。
- 按文件组织的 diff，每处改动标注类别（需求必需 / 安全必需 / 其他）。
- 业务影响（用用户行为描述，不用代码术语）。
- 回归测试清单。
- 额外发现和未修改声明。

**停下的信号**：方案里出现标为"其他"的改动但还放在 diff 里，或者"业务影响"用的是代码术语——要求 Codex 移出多余改动，用人话重写业务影响。

## 第四步：把方案发给 Claude 做正式终审

把 Codex 输出的完整方案复制给 Claude，然后使用 `prompts/zh-CN/11-final-plan-review.md`。

把模板的 `<PASTE_PLAN_HERE>` 替换为 Codex 给出的完整方案内容。

Claude 应该只指出真正会影响上线、偏离需求或扩大范围的问题，每条要么有具体指向，要么写"无"。

**停下的信号**：Claude 给了"建议加更多测试/日志/注释"这类泛泛建议，或者指出问题但不附具体位置——这些条目在后续处理时归档，不纳入方案。

## 第五步：把 Claude 的评审发回 Codex 复核

把 Claude 的完整输出复制给 Codex，然后使用 `prompts/zh-CN/12-review-response-triage.md`。

```text
下面是 Claude 对正式方案的批评。请不要盲从，把它转成下一版方案前的决策清单。

[Claude 的完整输出]

————————
[粘贴 prompts/zh-CN/12-review-response-triage.md 的完整内容]
```

Codex 的输出应该包含：

- 思路变化总结（逐项，不允许"整体符合预期"这类概括）。
- 对每条关键批评的复核结论，每条必须有代码证据、对话证据或产品语义证据。
- 采纳、部分采纳、不采纳、需要用户拍板的分类。
- 低价值条目的归档项。

**停下的信号**：Codex 没有验证就直接标"属实"或"不属实"——要求它先读对应文件，再给结论；没有打开文件就只能写"无法判断"。

## 第六步：执行确认后的方案

根据方案规模选择执行模板：

- **小方案**：在同一个 Codex 对话里直接发送 `prompts/zh-CN/07-small-execute.md` 的完整内容，不需要额外上下文。
- **大方案**：把最终确认的方案保存为本地文件，把 `prompts/zh-CN/08-strict-execute.md` 的完整内容发给 Codex，并把模板里的 `<PLAN_DOCUMENT_PATH>` 替换为实际文件路径。

Codex 的执行汇报应该包含：

- `git status` 检查结果。
- 改了哪些文件，对应方案哪条 diff。
- 跳过的 diff 及原因（diff 对不上、假设不成立等）。
- 编译 / 类型检查结果；如有自行修复的机械失误，单独列出。
- 自审：函数行为边界是否改变，调用方是否需要同步调整。
- 需要你手动确认的回归测试点。
- 未动区域确认。

**停下的信号**：汇报里出现方案以外的修改——停下，要求 Codex 回滚这些改动，只保留方案里列出的 diff。

## 常见错误

- 因为想法是自己提的，就要求 Codex 直接实现。
- 没读代码就讨论性能和权限。
- 路线没确认就进入方案。
- 方案没终审就执行。
- 把 Claude 的评审全盘采纳，或全盘否定，都没有验证。
- 执行时夹带未确认的改动。

## 最终你应该得到什么

- 对"当前不足是否属实"的代码证据。
- 对你初始想法的明确裁决。
- 一条被选中的实现路线。
- 一份经过终审和复核的方案。
- 一次严格按方案执行的改动。
