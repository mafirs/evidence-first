# Adversarial Review Flow Walkthrough

这个示例适合你已经拿到 Claude 的批评，但不确定该不该采纳时阅读。

示例里的产品、接口、文件名都是虚构的，只用于展示工作流。不要把真实业务数据、客户信息、密钥或私人对话提交进仓库。

## 虚构场景

你让 Claude 评审搜索功能的正式方案。Claude 给出这些批评：

```text
1. [核心] 方案没有验证权限过滤，可能把无权限文档搜出来。
2. [边缘] 搜索结果高亮没有设计，用户体验可能不好。
3. [凑数] 建议加更多日志和注释。
4. [核心] 方案直接重建索引，可能影响线上写入性能。
```

你不能直接全盘接受，也不能因为是 Claude 说的就全盘否定。你需要让 Codex 把这份批评转成下一版方案前的决策清单。

## 第一步：把 Claude 的评审发回 Codex 复核

把 Claude 的完整输出粘给 Codex，再复制 `prompts/zh-CN/review-response-triage.md`。

```text
下面是 Claude 对正式方案的批评。

原方案摘要：为文档搜索增加正文匹配能力，计划复用现有搜索 API，在后端扩展查询范围。

Claude 的评审内容：
1. [核心] 方案没有验证权限过滤，可能把无权限文档搜出来。
2. [边缘] 搜索结果高亮没有设计，用户体验可能不好。
3. [凑数] 建议加更多日志和注释。
4. [核心] 方案直接重建索引，可能影响线上写入性能。

本轮任务：不要盲从这份批评。请逐条复核，把它转成下一版正式方案前的决策清单。

————————
[粘贴 prompts/zh-CN/review-response-triage.md 的完整内容]
```

Codex 的输出应该包含：

- 思路变化总结（逐项，不允许"整体符合预期"这类概括）。
- 逐条回应，每条必须有代码证据、对话证据或产品语义证据。
- 每条的处理结论：采纳 / 部分采纳 / 不采纳 / 需要用户拍板。
- 低价值条目的归档项。

**停下的信号**：Codex 对第 1 条"权限过滤"说"属实"，但没有打开权限相关代码——停下，要求它先读对应文件，再给结论。没有打开过文件的，只能写"无法判断"，不能写"属实"或"不属实"。

## 第二步：判断回到哪个阶段

根据 Codex 的复核结论选择下一步：

- **批评推翻了原路线**（例如第 1 条"权限过滤"属实，且原方案没有覆盖）：回到 `prompts/zh-CN/route-with-user-idea.md` 或 `route-without-user-idea.md` 重新选路线。
- **路线仍然成立，方案需要补充约束**（例如需要加权限验证步骤，但整体路线不变）：回到 `small-plan.md` 或 `large-plan.md` 更新方案。
- **只有低价值条目**（例如第 3 条"建议加更多日志"）：归档即可，不要为了显得严谨而改方案。

**停下的信号**：因为看到 `[核心]` 标签就全部采纳，或者因为觉得 Claude 在凑数就全部跳过——两种都是错误。`[核心]` 仍然需要 Codex 验证，`[凑数]` 仍然需要看一眼确认确实是凑数。

## 第三步：把复核结论落实进下一版方案

把 Codex 的复核结论发给 Codex，明确指定哪些纳入、哪些不纳入，再用 large-plan 出新版方案。

```text
根据刚才的评审复核结论，请更新方案。

必须纳入：
- 权限过滤验证（第 1 条，复核属实）
- 避免影响线上写入性能（第 4 条，复核属实）

不纳入：
- 泛泛的"加更多日志和注释"（第 3 条，归档）

需要用户拍板：
- 搜索结果高亮是否进入本期范围（第 2 条，产品取舍，需要你决定）

————————
[粘贴 prompts/zh-CN/large-plan.md 的完整内容]
```

新方案应该包含：

- 已采纳批评对应的新 diff（例如权限验证的具体改动位置）。
- 不纳入的条目不出现在 diff 里。
- 用户拍板项单独列出，等待你决策后再处理。

**停下的信号**：新方案里出现了你明确说"不纳入"的改动，或者夹带了你没有确认的新约束——要求 Codex 移除这些内容，只保留你明确批准的部分。

## 第四步：需要时再次终审

如果新方案变化很大（例如权限验证涉及多个模块），把新方案再次发给 Claude，用 `prompts/zh-CN/final-plan-review.md` 再跑一遍终审。

如果只是小范围补充（例如只在已有权限检查后加一行过滤），你直接确认后进入执行即可，不必再跑一遍 Claude 终审。

## 常见错误

- 看到 `[核心]` 就全部采纳，没有让 Codex 验证。
- 看到 `[凑数]` 就完全跳过，漏掉被低估的问题。
- Codex 没读代码就说"属实"或"不属实"，你接受了这个结论。
- 把"加日志、加注释、更严谨"这类泛泛建议塞进方案。
- 给 Codex 说"综合考虑一下"，而不是明确指定哪些纳入、哪些不纳入。

## 最终你应该得到什么

- 一份思路变化总结。
- 一份逐条复核后的批评决策清单，每条有明确依据。
- 采纳、部分采纳、不采纳、无法判断的明确分类。
- 需要用户拍板的产品语义或工程取舍。
- 下一版方案里必须遵守的具体约束。
